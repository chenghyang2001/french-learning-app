# 任務清單：[功能名稱]

**輸入**：設計文件來自 `/specs/[###-feature-name]/`

**前置條件**：plan.md（必須）、spec.md（使用者故事必須）、research.md、data-model.md、contracts/

**測試**：以下範例包含測試任務。測試為**可選**——僅在功能規格明確要求時才加入。

**組織方式**：任務按使用者故事分組，使每個故事可獨立實作與測試。

## 格式：`[ID] [P?] [故事] 描述`

- **[P]**：可平行執行（不同檔案、無依賴關係）
- **[故事]**：此任務所屬的使用者故事（例如 US1、US2、US3）
- 描述中需包含確切的檔案路徑

## 路徑慣例

- **單一專案**：`src/`、`tests/` 位於 repo 根目錄
- **Web 應用程式**：`backend/src/`、`frontend/src/`
- **Mobile**：`api/src/`、`ios/src/` 或 `android/src/`

## 相依關係與執行順序

### 階段相依

- **初始化（第一階段）**：無依賴——可立即開始
- **基礎建設（第二階段）**：依賴初始化完成——阻塞所有使用者故事
- **使用者故事（第三階段起）**：全部依賴基礎建設完成
  - 之後可平行進行（若人力允許）
  - 或依優先級順序進行（P1 → P2 → P3）
- **修飾（最終階段）**：依賴所有目標使用者故事完成

### 使用者故事相依

- **使用者故事 1（P1）**：基礎建設（第二階段）後可開始——不依賴其他故事
- **使用者故事 2（P2）**：基礎建設（第二階段）後可開始——可能整合 US1 但應可獨立測試
- **使用者故事 3（P3）**：基礎建設（第二階段）後可開始——可能整合 US1/US2 但應可獨立測試

### 每個使用者故事內部

- 測試（若包含）必須先寫並確認失敗，才能開始實作
- 模型先於服務
- 服務先於端點
- 核心實作先於整合
- 一個故事完成後再進入下一個優先級

### 平行執行機會

- 所有標記 [P] 的初始化任務可平行執行
- 所有標記 [P] 的基礎建設任務可平行執行（第二階段內）
- 基礎建設完成後，所有使用者故事可平行開始（若人力充足）
- 一個故事內所有標記 [P] 的測試可平行執行
- 一個故事內標記 [P] 的模型建立任務可平行執行
- 不同使用者故事可由不同開發者平行推進

## 實作策略

### MVP 優先（只做使用者故事 1）

1. 完成第一階段：初始化
2. 完成第二階段：基礎建設（關鍵——阻塞所有故事）
3. 完成第三階段：使用者故事 1
4. **停下來驗證**：獨立測試使用者故事 1
5. 若就緒則部署／展示

### 漸進交付

1. 完成初始化 + 基礎建設 → 基礎就緒
2. 加入使用者故事 1 → 獨立測試 → 部署／展示（MVP！）
3. 加入使用者故事 2 → 獨立測試 → 部署／展示
4. 加入使用者故事 3 → 獨立測試 → 部署／展示
5. 每個故事在不破壞既有故事的前提下增加價值

### 團隊平行策略

有多位開發者時：

1. 團隊共同完成初始化 + 基礎建設
2. 基礎建設完成後：
   - 開發者 A：使用者故事 1
   - 開發者 B：使用者故事 2
   - 開發者 C：使用者故事 3
3. 各故事獨立完成並整合

## 注意事項

- [P] 任務 = 不同檔案、無依賴關係
- [故事] 標籤將任務對應到特定使用者故事，便於追蹤
- 每個使用者故事應可獨立完成並測試
- 實作前確認測試失敗
- 每個任務或邏輯群組完成後提交 commit
- 在任何檢查點停下來獨立驗證故事
- 避免：模糊任務、同一檔案衝突、破壞獨立性的跨故事依賴
